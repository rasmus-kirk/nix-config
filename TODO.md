# TODO

## Wrap config-leaf tools as flake packages

**Idea:** Replace the home-manager-writes-config-files pattern with
flake packages that bundle each tool + its config and expose a wrapped
binary. Same config that runs on every machine you own becomes
`nix run github:rasmus-kirk/nix-config#<tool>` from anywhere.

### Motivation

Today every tool's config lives in `modules/home-manager/<tool>/default.nix`
and is materialized into `~/.config/<tool>/` by home-manager switch. That
requires HM installed on every target machine and tied into a NixOS-style
flow.

A wrapped-package equivalent (e.g.
`pkgs.helix-rk = pkgs.symlinkJoin { ... wrapProgram --set XDG_CONFIG_HOME ${configDir} }`)
ships the binary + config as a single self-contained derivation. Works
on any machine with nix installed.

### Scope (what wraps cleanly, what doesn't)

**Wrap-friendly** (config via `--config` / `--config-dir` / `XDG_CONFIG_HOME`,
no machine-specific state):

- `helix` — `XDG_CONFIG_HOME` override + LSPs in PATH (closure pain — see below)
- `foot` — `foot --config /nix/...`
- `yazi` — `yazi --config-dir /nix/...`
- `mpv` — `mpv --config-dir /nix/...`
- `mvi` — wraps mpv → same mechanism
- `zathura` — `zathura --config-dir /nix/...`
- `zsh` — `ZDOTDIR=/nix/... zsh`, bundle plugins (gruvbox-powerline etc.)
- `jiten` / `monero` — verify per-tool flags
- `git` — `GIT_CONFIG_GLOBAL=/nix/...`, but the embedded `signKey` path
  (`/data/.secret/...`) only resolves on machines that have it

**Wrap-resistant** (system integration / per-user state / not a single
binary):

- `chromiumLaunchers` — `.desktop` files in `~/.local/share/applications/`
- `xdgMime` — `~/.config/mimeapps.list`
- `userDirs` — `xdg-user-dirs-update` writes to `$HOME`
- `fonts` — fontconfig system caches
- `gruvboxTheme` — system-wide GTK/QT theme env
- `ssh` — wrappable via `-F`, but known_hosts is mutable user state and
  IdentityFile path is per-machine
- `homeManagerScripts` / `scripts` — these *are* scripts; nothing to wrap
- `terminalTools` — just `home.packages` lists; nothing to wrap
- `stateBackup` — systemd unit
- `box` — already a custom launcher; different pattern entirely

So roughly half of the ~22 HM modules are wrap candidates; the rest are
the system-integration glue that justifies HM existing.

### Gotchas to plan for

1. **LSPs and runtime deps balloon the closure.** Wrapping `helix` to
   bundle rust-analyzer + omnisharp-roslyn + harper + gopls + marksman +
   lldb means the wrapped binary pulls in dotnet, llvm, several language
   toolchains. Either bundle (huge), skip (lose 80% of config value), or
   mix (brittle line to draw). Decide per-tool.

2. **Env overrides leak to child processes.** `wrapProgram --set
   XDG_CONFIG_HOME` is inherited by every LSP/subprocess helix spawns.
   Usually they fall back to defaults; occasionally surprising.

3. **Per-user / per-machine state can't be baked in.** Git signing key
   path, gh auth, ssh known_hosts, browser profiles — these live in `$HOME`
   and aren't reproducible. Wrapped packages can reference these by path
   but only work on machines where the path exists.

4. **System wiring still needs HM.** `$EDITOR`, default applications,
   `.desktop` entries, xdg-open associations, theme env. The wrapped
   pattern doesn't subsume these — they continue to live in HM regardless.

### Proposed refactor shape

For each wrappable tool:

1. Move the config definition out of the HM module into
   `lib/<tool>-config.nix` (or similar) — one source of truth as
   plain TOML/text generated by Nix.
2. Add `flake.packages.${system}.<tool>-rk` that builds the wrapped
   binary by `symlinkJoin` + `wrapProgram`, consuming the config from
   step 1.
3. Slim the HM module to just `home.packages = [ self.packages.${system}.<tool>-rk ]`
   plus any `$EDITOR` / xdg-open / desktop wiring that still has to live
   in HM.

Result:

- On your own machines: HM stays in charge of system integration, but
  delegates "what config does helix use" to the wrapped package.
- On any other nix machine: `nix run github:rasmus-kirk/nix-config#helix-rk`
  gives you your editor with the right keymaps + theme. No HM, no
  switch, no /data assumptions (except for the per-machine stuff like
  the YubiKey signing key — those degrade gracefully or fail loud).

### Proof-of-concept first

Don't refactor all of them at once — but **pick `helix` as the first
PoC, not `yazi`**. The temptation is the opposite (start with the
easy one, ride momentum to the hard ones), but that path gives you
false confidence: yazi wrapping is essentially `--config-dir` + a
symlink, and proving it works tells you nothing about whether the
pattern actually survives the cases that matter.

helix is where the real questions live:

- **LSP closure decision** — bundle / skip / mix? Has to be answered
  before the pattern can claim "works for editors." If the answer is
  "skip, document," that's fine, but it has to be the answer
  deliberately.
- **`XDG_CONFIG_HOME` override leaks to subprocesses** — helix spawns
  LSPs that inherit the env. Yazi spawns nothing that cares; doesn't
  exercise the gotcha.
- **`programs.helix.defaultEditor`** — what replaces the home-manager
  sugar that sets `$EDITOR`? Yazi has no equivalent; the system-wiring
  question doesn't come up.
- **TOML emission for nested settings** — helix's `keys.normal` /
  `keys.insert` and `languages.language[]` are far chunkier than
  yazi's config. If the refactor's "shared config-as-Nix-file"
  story works for helix, it works for everything smaller.

If helix lands cleanly, the remaining wrappable tools (yazi, foot, mpv,
zathura, etc.) are mechanical applications of the same shape. If helix
*doesn't* land cleanly, you find that out before refactoring 8 other
modules and walking them all back.

Verify on the PoC:

- `nix run .#helix-rk` works on a fresh machine (or in an unrelated
  nix shell).
- The home-manager `kirk.helix` module reduces to "install the wrapped
  package + set `$EDITOR` / any other HM-only bits."
- LSPs in `home.packages` still get discovered by the wrapped helix
  on YOUR machines (via the user's normal PATH), even though they're
  not bundled in the wrapper itself.
- The `XDG_CONFIG_HOME` override doesn't visibly break rust-analyzer
  or other LSPs spawned by helix.

### Decision needed before starting

For each wrappable tool: **bundle runtime deps in PATH, or assume user
provides them on the target machine?** Defaults to "assume" probably,
with documentation that ports work but LSP/preview features need the
user to have those tools installed. For tools where the runtime deps
are tiny (mvi's mpv plugins) — bundle. For helix-with-LSPs — don't.
